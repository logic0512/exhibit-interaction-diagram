# 展项装置交互图

## 核心定位

把展项、装置或多媒体互动创意，转成一张清晰能看懂玩法的装置交互图。它既可以直接生图，也可以先辅助扩展想法、补全玩法，再生图。

最高标准：没参与过讨论的人看到图后，能在 3 秒内理解“人怎么参与、触发了什么、反馈在哪里发生、这个互动让人理解了什么内容关系”。

## 先读这些参考

按任务需要读取，不要一次塞满上下文：

- `references/human-component.md`：每张图必须有的人物组件、动作、尺度和禁忌。生成前必须读。
- `references/idea-expansion.md`：把抽象主题、内容方向或模糊创意扩展成可画的互动玩法。创意不清时必须读。
- `references/spatial-fit.md`：根据空间大小、亮度、观看距离、人流和安装条件判断载体是否合适。涉及空间、投影、大屏、灯光、儿童尺度或装置尺寸时必须读。
- `references/hardware-grounding.md`：根据展陈硬件判断规则校准设备类型、安装方式、工作距离、观看距离、维护空间和感应范围。涉及投影、屏幕、LED、触控、感应、声音、机柜或具体硬件时必须读。
- `references/carrier-components.md`：屏幕、投影、灯光、感应、音响、操作台、机械件等载体的可识别表达。创意涉及相应载体时读取。
- `references/information-content.md`：设计屏幕、投影、灯光、声音、机械反馈上的交互信息内容。创意涉及媒体反馈画面时必须读。
- `references/interaction-logic.md`：把创意转成“人 - 触发 - 反馈 - 结果”的画面逻辑。
- `references/composition-patterns.md`：选择单张图的主构图模式。生成前必须选 1 种，不要混合多个模式。
- `references/visual-dna.md`：展项方案草图的视觉边界和禁忌。
- `references/prompt-brief.md`：生图前的结构化判断母版，用来想清楚玩法、空间、硬件、内容和参考资产。
- `references/runtime-prompt.md`：最终喂给生图工具的短提示词母版，只带入当前命题相关规则。
- `references/qa-checklist.md`：生成后检查和迭代规则。
- `references/validation-rubric.md`：修改 skill、复盘失败或跑测试样例时使用的评分表。
- `references/visual-system.md`：内置 atomic 视觉资产的使用方式。需要校准人物、动作、载体、结果或交互链路时读取。
- `assets/visual-system/atomic/`：人物、动作姿态、载体识别、结果反馈、交互链路的单意图视觉资产。每张图只能对应一个意思，不能使用组图作为生图参考。
- `references/example-gallery.md` 和 `assets/examples/`：完成图 examples。只作低频视觉校准，帮助判断某类设备交互的完整图面样式；不要复制构图、设备造型或主题内容。
- `test-prompts.json`：回归测试样例。修改 skill、assets、examples、提示词模板、空间/硬件/QA 规则后使用。

## 工作流

### 1. 输入分流

先判断用户当前处在哪个状态：

- 直接生图：用户已经给出明确玩法，并且能说清参与者、动作、触发、反馈、内容变化、装置/空间主体。不要长篇讨论，直接提炼玩法并生图。
- 辅助表达：用户有概念或装置方向，但人物动作、触发或反馈不清。先读 `idea-expansion.md`，补全 1 个清晰方案，再进入生图。
- 创意扩展：用户只有主题、内容、目标人群或空间诉求。先读 `idea-expansion.md`，给 2-3 个互动方向，并推荐最适合画成图的一个方向。

直接生图前必须过“可画性门槛”：能用一句话说清“观众通过某个动作，改变/选择/连接/拆解/推进某个内容对象，从而理解某个主题关系”。如果只能说清“做动作后出现反馈”，但说不清内容为什么必须这样互动，先进入辅助表达，不要调用生图。

还必须过“交互装置定义门槛”：装置不是承载信息的容器，而是观众可操作、会响应、能让信息通过操作被获得的媒介。如果主题信息、参数、原理拆解或说明内容只是摆在装置表面，去掉人物动作后仍然完整成立，这不是合格互动装置，要先把信息转成观众可执行的任务、可改变的内容变量和即时反馈。

如果缺少参与者、动作、触发、反馈、结果里的 3 项以上，不要急着生图。若载体选择强依赖空间条件，或媒体反馈强依赖主题内容，也要先补全空间假设和反馈内容。若用户只给出主题、空间、人群和“希望多人/沉浸/互动”这类诉求，默认不是可直接生图需求，先补出玩法机制。

### 2. 理解创意

先把用户的想法压缩成一句玩法判断：

> 参与者在某个空间里，通过某个动作操作某个装置，改变某个内容变量，并在某个反馈区域看到/听到/触到对应变化，从而理解某个主题关系。

然后按需读取 reference，不要把所有规则同时塞进上下文：

- 创意抽象或机制不清：读 `idea-expansion.md`，先把主题翻译成动作。
- 玩法链路、多人任务、内容动作关系不清：读 `interaction-logic.md`。
- 反馈画面、屏幕/投影/灯光/声音/机械内容不清：读 `information-content.md`。
- 涉及空间大小、亮度、墙面留白、沉浸空间、儿童尺度或装置尺寸：读 `spatial-fit.md`。
- 涉及投影、屏幕、LED、触摸、感应、声音、机柜或具体硬件：读 `hardware-grounding.md`。
- 需要让屏幕、投影、灯光、感应、音响、操作台或机械件更可识别：读 `carrier-components.md`。
- 生成前选择 1 个主构图模式：读 `composition-patterns.md`。

核心判断只保留三条：

- 创意驱动生成，组件只负责让载体可识别，不从组件库反推创意。
- 信息必须通过观众操作被揭示、改变、比较、验证、推进或汇总，不能把信息贴在装置上冒充互动。
- 空间、硬件、人体尺度和装置形态必须落地；若任何一项强依赖未知条件，先补出保守假设或进入辅助表达。

### 3. 选择必要画面要素

人物是强制组件，每张图都要有人物，而且人物必须参与核心玩法。人物不是普通提示词描述，而是本 skill 的硬锁视觉层：生成前必须从 `assets/visual-system/atomic/human/` 选择并查看 1 张最接近当前人群和动作的人物资产，作为所有人物的统一视觉参考。

人物参考优先级高于完成图 example、主题内容和设备造型。成人、儿童、亲子只能改变身高、姿态和数量，不能改变角色画法。不要让模型自由生成发型、五官、裙装、背包、复杂衣服、写实背影或不同人物风格。

其他载体按需调用：如果创意涉及屏幕、投影、灯光、感应器、音响、地面点位、操作台、机械件等，就优先用 `carrier-components.md` 的识别锚点表达。没有涉及就不要硬塞。

内置资产不是复制模板。屏幕、投影、音响、操作台、机械件都可以有造型变化；稳定的是它们让人一眼可识别，并且和人物动作、触发、反馈关系清楚。

如果创意需要内置资产没有的新装置，可以生成新造型，但必须沿用同一套人物尺度、交互关系、反馈标注和展项草图语言。儿童或亲子展项要明确体现儿童身高、眼高、手可达范围和成人陪伴关系。

### 4. 单张生成

如果用户明确要求“生成 / 画 / 出图 / 做一张图 / 帮我把创意变成图”，不要先输出长篇分析。读取必要参考后，使用内置 `image_gen` 生成一张图。

如果任务涉及人物尺度、儿童视角、具体动作姿态、投影/屏幕/音响等载体识别、结果反馈表达，或上一次出图风格漂移，先查看 `assets/visual-system/atomic/` 里对应的单意图资产。

人物资产必须每次都选，不是可选项：

- 亲子、儿童尺度、空间尺度：优先 `assets/visual-system/atomic/human/01-adult-child-scale.png`
- 儿童低位操作、桌面、按钮、地面点位：优先 `assets/visual-system/atomic/human/02-child-low-reach.png`
- 墙面触摸、立面屏幕、近墙交互：优先 `assets/visual-system/atomic/human/03-touch-wall.png`

在调用生图前，必须把选中的人物资产作为可见参考，并在提示词里明确说明：所有人物必须沿用该参考的人物 IP，只允许改变身高、姿态、朝向和数量。

如果是在同一个项目、同一个主题或同一张图基础上继续调整，先建立“迭代锁定”：沿用上一版已经成立的视觉 DNA、人物资产、人物画法、空间假设、主构图模式、色彩策略和装置尺度。除非用户明确要求改变风格、构图或设备形态，否则下一版只能修正问题点，不能把方案从手绘展项草图漂移成写实渲染、暗场舞美、PPT 信息图、UI 看板或另一套人物系统。

每次只选择和当前玩法直接相关的少量 atomic 资产：通常 1 张人物或动作、0-1 张载体、0-1 张结果反馈。交互链路先用 `interaction-logic.md` 和提示词控制，不使用步骤条图片。不要把整套资产一次性塞进提示词，不要使用组图、contact sheet、退稿图或派生过程图作为最终生图参考。

如果当前玩法明确属于某类设备交互，或需要校准完成图质量，可以查看 `assets/examples/` 中最接近的 1 张完成图 example。examples 只用于校准整体图面密度、人物/设备/反馈关系和设备类型可读性；不要复制 example 的构图、设备造型、标注位置或主题内容。

examples 不能覆盖人物 atomic 资产。若 example 与人物资产在人物画法上冲突，以人物 atomic 资产为准。

调用生图前，必须先用 `references/prompt-brief.md` 完成一次结构化判断，再用 `references/runtime-prompt.md` 组装最终提示词。不要把 brief、QA、载体规则全文塞进最终提示词；最终提示词只保留当前命题真正相关的规则和约束。

如果 `prompt-brief.md` 里的可画性门槛没有通过，停止生图，先用一句清晰方案补齐“参与动作 - 内容变量 - 反馈变化 - 体验结果”。只有当这条链路成立后，才进入 `runtime-prompt.md`。

最终提示词使用 `runtime-prompt.md` 组装：

- 每次都使用 `Base Prompt`。
- 按命题只追加 1-4 个 `Conditional Blocks`，例如儿童、多人、投影、IoT、空间场景、机械控件、穿戴设备。
- `Repair Prompts` 只在失败后使用，不进入首次生图 prompt。
- 不要把 QA checklist、brief、reference 或 runtime 全文塞进 image model。

每次默认只生成一张图。只有用户明确要求多方案或多图时，才输出多张。

### 5. 检查与迭代

生成后按 `qa-checklist.md` 检查。如果玩法不清楚，优先重生成或局部编辑，而不是用文字解释弥补。

常见失败要立刻修：

- 人只是站在旁边，没参与玩法。
- 人物出现发型、五官、写实服装、裙装、背包、不同角色气质，或与人物 atomic 资产不一致。
- 同一张图里成人、儿童、旁观者像不同插画系统。
- 设备与目标面距离不成立，例如探照灯/投影/扫描灯贴着墙却声称照到墙上。
- 人物够不到操作点，或手臂、站位、儿童身高和控制点高度不成立。
- 把手、手轮、拉杆、滑块、按钮、道具或设备部件看起来接在人身上，缺少连接到装置的底座、转轴、支架或安装面。
- 投影源有多个互相冲突的发光口，例如前镜头发光但光束从顶部槽口出来，导致无法判断真实光路。
- 普通交互被画成顶天立地轨道、全高立柱、龙门架、吊架或大型桁架，导致装置过重并破坏空间秩序。
- 硬件类型与空间不匹配，例如明亮空间普通投影、近距离粗颗粒 LED、无拼缝拼接屏、无识别范围的传感器、无听音位置的音响。
- 触发动作和反馈区域没有视觉连接。
- 内容和交互没有因果关系，只能看出“触发后有反馈”，但看不出为什么这个动作帮助理解主题。
- 把信息放在装置上冒充互动：去掉人物动作后，主题信息、参数或原理说明仍然完整成立，像给设计者看的说明展板。
- 过程型主题没有同一内容对象的变化链，变成多个无关点位亮起。
- 选择、识别、复原、比较或时间推进只画了一个静态对象，导致玩法说不清。
- 同一项目迭代时人物、线稿、色彩、空间质感或图面语言漂移，像换了一套 skill。
- 组件堆太多，像设备清单。
- 屏幕、投影、感应、信息层全部变成蓝色，画面缺少主题区分和材质层次。
- 画成商业海报、舞美效果图、PPT 或 UI 设计稿。
- 只好看，但看不懂怎么玩。

如果是在修改或打磨本 skill，而不是单次生图，必须使用 `test-prompts.json` 做回归检查：

- 小改动至少 dry-run 3 个核心案例：`T01_tree_spotlight_lightweight_regression`、`T02_yangtze_fishing_ban_table`、`T08_abstract_concept_to_action`。
- 涉及人物、空间、硬件、载体、examples、prompt brief 或 runtime prompt 的改动，按 `references/validation-rubric.md` 评分。
- 低于 80 分不要视为通过；低于 70 分说明该案例失败，需要回到规则、资产或提示词修正。
- 失败案例要记录为新的测试项或补进现有测试项，避免同类问题再次出现。

## 输出口径

按输入分流决定输出：

- 直接生图：生成前只保留必要的一两句话说明，不输出完整分析。
- 辅助表达：先给 1 个补全后的清晰互动方案，说明“参与动作 - 触发 - 反馈 - 结果”，再生图或询问是否生图。
- 创意扩展：给 2-3 个互动方向，并推荐最适合画成装置交互图的一个方向；用户确认或意图明显时再生图。

生成后简短说明：

- 图表达的玩法
- 哪个动作触发了哪个反馈
- 反馈里呈现的核心内容信息
- 使用的空间假设或载体尺度判断
- 如有保存路径，给出路径

不要长篇解释风格理论。扩展阶段服务于选方向，生图阶段让图负责传达玩法。
