# Prompt Brief

这个文件不是最终生图提示词。它用于在生图前把创意判断清楚，避免把所有规则一股脑塞进 image model。

最终喂给生图工具的内容，应从本 brief 中提取到 `runtime-prompt.md`，只保留当前命题真正相关的部分。

## 使用时机

- 直接生图：只在核心链路已经成立时快速填写已知字段；只能用保守假设补齐非核心细节，例如空间亮度、画幅或装置材质，不要补齐核心玩法后直接生图。
- 辅助表达：先用 `idea-expansion.md` 把抽象主题转成动作，再把补全方案交给用户确认；确认前不要进入最终生图 prompt。
- 回归测试：记录本案例使用了哪些空间、硬件、人物和载体约束。

## 0. Pre-Generation Readiness Gate

生图前先判断输入是否已经够画。不要因为用户说“我想做一个互动展项”就直接进入生图。

### 玩法与内容是否成立

- 参与者是谁，人数和身高是否影响动作。
- 参与者做什么具体动作。
- 这个动作操作了哪个内容变量：时间、位置、对象、层级、连接、参数、选择、顺序、拆解、重组或协作状态。
- 反馈在哪里发生，反馈内容如何随动作变化。
- 观众通过这个变化理解了什么主题关系。
- 观众是否必须通过操作才能获得这条信息；去掉人物动作后，信息是否还完整成立。
- 过程型主题是否有同一个内容对象的变化链。
- 选择、识别、复原、比较或时间推进是否有足够对象、状态或节点。

如果只能回答“触发后出现反馈”，但不能回答“内容为什么通过这个动作被理解”，不要生图。先回到辅助表达，补出一个清晰的玩法机制。

如果只能回答“装置上展示了哪些知识点/参数/说明”，但不能回答“观众如何操作并获得这些信息”，也不要生图。先把信息转成可玩的任务和即时反馈，避免把静态信息摆放误当成互动装置。

### 用户确认门槛

如果参与者、动作、触发、反馈、结果、内容变量中缺少 3 项以上，或主题只表达“想介绍/了解/展示某类内容”，必须先输出方向或补全方案，不调用生图工具。

补全方案要明确哪些是用户已给信息，哪些是推断假设。然后用一句话询问用户是否按该方向生成，或让用户选择方向。

只有三种情况可以越过确认直接生图：

- 用户输入本身已经满足直接生图门槛。
- 用户明确说“按你的判断直接生成 / 直接测试 / 不用讨论先出一版”。
- 当前任务是回归测试或随机测试，且你已标注测试模式、缺失项、补全假设和本次验证目标。

### 空间与硬件是否成立

- 装置主体属于哪种形态：独立轻量、家具化、墙体局部、地面局部、展柜结合、空间结构结合、特殊造型设备、手持/穿戴或大型结构。
- 空间亮度、尺寸、观看距离是否支撑载体选择。
- 成人或儿童是否自然够得到操作点。
- 设备工作距离是否成立。
- 如果使用投影，投影类型、镜头朝向、目标介质、光锥中心线、落点和人物遮挡关系是否成立。
- 如果是空间场景或沉浸空间，是否能表现房间边界、地面范围、通行路径和人物站位。

### 视觉与引用是否成立

- 是否选择了必要的人物 atomic 资产。
- 是否只选择了当前命题必要的载体或结果参考。
- 色彩是否避免蓝色过量，并根据主题使用非蓝色辅助信息层。
- 同项目迭代是否保持上一版已成立的人物、线稿、材质、空间和色彩锚点。
- 是否避免用户未指定的真实品牌、地标和专属建筑。

## 2. Interaction Chain

- Participant:
- Core action:
- Trigger:
- Feedback area:
- Result:
- Content variable controlled by the action:
  - 时间阶段 / 空间位置 / 对象选择 / 层级显现 / 关系连接 / 参数变化 / 顺序推进 / 拆解重组 / 协作汇总 / 其他。
- Content-interaction thesis:
  - 观众通过这个动作理解了什么内容关系。
- The visual link that must be readable:
  - 人的动作 -> 触发点/感应范围 -> 反馈区域 -> 内容变化。
- Gameplay evidence:
  - 画面中用什么对象、状态、阶段或位置证明玩法成立。
  - 选择、比较、识别、复原、拆解、分层或时间推进类玩法，通常需要 2-4 个对象/状态/节点；不要只画一个静态物件。

如果这条链路无法一句话说清，先不要进入生图。

## 3. Information Content

- Theme/content:
- Feedback content:
  - 屏幕、投影、灯光、声音、气味或机械反馈中具体出现什么。
- Content change:
  - 互动前后内容如何变化。
- State count:
  - 需要几个内容状态才能看懂，例如未触发/已触发、选中/未选中、表层/内部层、过去/现在、整体/拆解。
- Content-carrier pairing:
  - 为什么这个内容适合当前载体。
  - 如果使用投影，为什么这个内容适合投到当前介质，而不是独立屏幕或右侧幕布。
- Content-action pairing:
  - 为什么当前动作适合表达这个内容，而不是普通按钮触发。
  - 动作是否改变了内容变量，而不是只打开内容。
- Interaction-obtained information:
  - 哪些信息必须通过观众操作才显现、变化、比较、验证或汇总。
  - 如果去掉人物动作，这些信息是否还完整成立；若成立，说明当前设计偏静态展板，需要改成可操作任务。
- Process object:
  - 如果是演化、发展、传播、流转、转化或系统协作，是否有同一个内容对象贯穿多个阶段，例如同一条消息、同一件展品、同一条路线、同一组数据或同一个模型。
- Specificity boundary:
  - 用户没有指定真实城市、建筑、品牌、地点、人物或车辆品牌时，使用泛化轮廓、编号对象和抽象信息。
  - 不要自行生成真实地标、品牌标识、可识别专属建筑或具体车型品牌。

媒体反馈不能是固定蓝色 UI、空白投影或无意义发光效果。它必须服务当前主题和玩法。

色彩也不能固定成蓝色逻辑。生图前先决定：

- 蓝色/青色是否真的需要作为投影、屏幕、感应或数字反馈。
- 除蓝色外，主题内容使用什么克制辅助色，例如生态绿色、历史琥珀、机械暖灰、警示红橙、记忆黑白照片调或淡紫。
- 蓝色/青色面积是否被限制在局部反馈层，而不是覆盖整张图。

## 4. Spatial And Hardware Grounding

- Space condition:
  - 尺寸、亮度、人流、通道属性、儿童空间、黑场/半暗/明亮等。
- Carrier suitability:
  - 当前载体为什么适合这个空间。
- Device form:
  - 独立轻量 / 家具化 / 墙体局部 / 地面局部 / 展柜结合 / 空间结构结合 / 特殊造型设备 / 手持或穿戴 / 大型结构。
  - 普通互动默认优先独立轻量、家具化、局部安装或空间融合式装置。
  - 不要把装置默认理解成展台；如果内容和动作更适合墙面、地面、展柜、通道、转角或特殊造型设备，优先使用对应形态。
  - 顶天立地轨道、龙门架、全高立柱、天花吊架、大型桁架只在明确需要时使用。
- Device working distance:
  - 设备到墙面、地面、物体、屏幕或感应目标之间的合理距离。
  - 投影、探照灯、扫描灯、定向声源、体感感应必须画出源头、路径或检测范围。
  - 投影必须先判断类型：正投、短焦/超短焦、背投、桌面投影、模型投影、隐藏投影源或投影融合。
  - Projection target fit:
    - 目标介质类型：物体/展品表面、桌面/沙盘、装置本体、半透明屏/纱幕、地面/路径、墙面/幕布、背投/内嵌投影、其他。
    - 为什么这个介质最适合当前玩法和内容：
    - 是否可以避免外接幕布/右侧屏幕：
  - Projection optical axis:
    - 投影源位置：
    - 唯一有效出光口：
    - 镜头朝向：
    - 目标介质：
    - 目标面位置：
    - 投射距离：
  - 每个投影源只能有一个有效出光口。普通前置镜头、顶部反射窗、斜面出光槽、墙面开口或隐藏孔位只能选一个；光束必须从该口径出发。
  - 投影目标面必须位于镜头前方的光锥内；不要为了满足“有投影内容”而默认放到右侧墙面。
  - 如果镜头朝向、光锥中心线和目标面不能在画面里共线或近似共线，优先改构图、改投影类型、改目标介质，或换成屏幕/实体反馈。
  - 如果外部墙面/独立幕布不是内容表达的最佳介质，不要为了光轴清楚而采用“左装置 + 右幕布”构图。
  - 普通正投不能贴着目标面；短焦/超短焦可以近距离，但必须表现机身、支架、倾斜光路和落点方向。
  - 投影光路必须从投影源出发，不得和人物手势、按钮箭头或交互联动线混在一起。
- Human reach envelope:
  - 人站在哪里，手、脚、视线是否自然够得到。
  - 儿童展项要检查儿童手高、眼高、步幅和成人陪伴关系。
  - 如果设备需要在高处或远处移动，必须有可触达的把手、滑块、拉绳、手轮、平衡臂或低位控制件。
- Human-device boundary:
  - 把手、手轮、拉杆、滑块、按钮、道具、支架或设备部件分别锚定在哪里。
  - 这是固定展项硬件、手持设备，还是穿戴设备。
  - 固定硬件是否错误地让人手/身体承载；穿戴或手持设备是否有真实绑带、手柄、衣物模块或固定结构。
  - 控制件是否有可读的转轴座、安装面、轨道、支架、台座面板、墙面安装点、地面底座或道具本体。
  - 固定硬件的人体轮廓和控制件固定端是否分开；儿童身体不能成为隐藏支架。
- Hardware category:
  - 投影机 / 短焦投影 / 背投 / LCD 商显 / 拼接屏 / LED 点阵 / 透明 LED / 地踩 LED / 触摸屏 / 红外框 / 激光雷达 / Kinect / 编码器 / 音响 / 机柜 / 自定义硬件。
- Installation method:
  - 吊装 / 壁挂 / 落地支架 / 桌内嵌入 / 墙后 / 屏幕后 / 地面嵌入 / 设备机身集成 / 其他。
- Physical cues:
  - 边框、拼缝、箱体厚度、支架、吊点、维护门、散热、机柜、传感器视场、投影锥、声场方向。

- Optional action reference:

## 6. Rules To Inject

只选择当前命题相关规则，不要全量复制。

- Human lock:
  - 所有人物必须沿用选中人物资产的统一 IP。
  - 只能改变身高、姿态、朝向和数量。
  - 固定展项硬件不能由人物承载；VR、手持控制器、穿戴传感器或互动服只有在用户明确要求时允许，并必须画出穿戴/握持结构。
- Carrier rules:
  - 屏幕：有清晰显示面、厚度或边框，内容服务主题。
  - 投影：有投影源、路径、目标面，亮度和空间成立。
  - 投影出光口：每个投影源只有一个有效出光口，光束从该口径发出，不让前镜头和顶部槽口同时发光。
  - 投影方向：先锁定镜头朝向，再沿镜头方向放置目标面；光路从源头到落点，人物不挡光；普通投影需要可见距离，近距离投影必须明确短焦/背投/嵌入方式。
  - 投影目标：先选择最适合内容的目标介质，优先物体、模型、桌面、装置本体、地面、半透明介质或背投；墙面/独立幕布只在确实需要大画面、群体观看或远距离阅读时使用。
  - 灯光：有灯具、照射范围、被照亮的内容变化。
  - 感应：有识别区域、传感器位置、身体进入方式。
  - 声音：有声源、听音位置、声场方向。
  - 气味：有闻香口、气味模块或样本来源，气味范围局部且和主题信息绑定。
  - 机械：有可操作件、固定端/转轴座/安装面、运动方向、反馈结果。
- Spatial rules:
  - 小空间避免大型展台和沉浸式投影。
  - 明亮空间避免普通投影作为主反馈。
  - 普通手动交互避免重型全高结构。
- Content rules:
  - 反馈画面必须有主题信息。
  - 主题信息不能预先完整贴在装置上。它必须通过观众动作被揭示、改变、比较、验证或汇总。
  - 不把小屏幕改成独立大屏、浮动说明卡或底部说明条，除非用户明确要求。
  - 不生成底部步骤条、总结说明框、完整流程分镜或教学海报，除非用户明确要求 storyboard。
  - 选择/比较/识别/复原/时间推进需要足够对象、状态或节点支撑。
  - 蓝色/青色只作为局部数字反馈色，不把所有信息层都画成蓝色。

## 7. Runtime Prompt Assembly

进入 `runtime-prompt.md` 时只带入这些内容：

- 一句话玩法。
- 人 - 触发 - 反馈 - 结果链路。
- 内容变量和内容交互论点。
- 反馈中的具体信息内容。
- 如果是过程型主题，说明贯穿全图的同一个内容对象。
- 玩法证据：对象数量、状态数量或连续位置。
- 当前空间和硬件约束。
- 人机边界：控制件锚点、手部握持关系、人体和设备固定端的分离。
- 投影类型、镜头朝向、目标介质、光学轴线和可实现距离。
- 投影目标介质为什么适合当前内容，是否避免了同质化右侧幕布构图。
- 色彩策略：主反馈色、辅助强调色、蓝色面积限制。
- 选中的人物资产。
- 0-2 个必要载体或结果参考。
- Base Prompt。
- 当前命题相关的 1-4 个 Conditional Blocks。

不要把本文件、QA checklist 或 reference 整段复制进最终提示词。最终提示词应该短、准、可执行。
