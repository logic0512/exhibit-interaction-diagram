# 验证评分表

## 作用

本文件用于测试和复盘 `exhibit-interaction-diagram` 的输出是否稳定。每次修改 skill、资产、examples、提示词模板或硬件/空间规则后，用 `test-prompts.json` 里的案例生成或 dry-run，再按本表评分。

不要只看图是否漂亮。这个 skill 的合格标准是：一张图能清楚表达展项玩法，并且装置在空间、硬件、人体尺度上成立。

## 使用方法

每个测试案例给 0-100 分。低于 80 分必须重写提示词或规则；低于 70 分说明该案例失败，不能把这轮改动视为通过。

评分时先写一句结论：

```text
Case {id}: pass / partial / fail, score {n}/100, blocking issues: {最多3条}
```

## 评分维度

| 维度 | 分值 | 判定重点 |
|---|---:|---|
| 玩法链路清晰度 | 20 | 3 秒内能看出人做什么、触发什么、反馈在哪里、结果是什么 |
| 人物 IP 与动作 | 15 | 人物统一、无发型五官、成人/儿童尺度合理、主人物参与核心动作 |
| 空间匹配 | 15 | 空间大小、亮度、观看距离、人流和装置尺度匹配 |
| 硬件落地 | 15 | 投影/屏幕/LED/感应/声音/机械设备有真实工作关系，控件有清楚锚点 |
| 装置形态与空间秩序 | 15 | 普通互动优先独立轻量、家具化或局部安装；大型结构有明确理由 |
| 反馈内容信息 | 10 | 屏幕/投影/灯光/声音/机械反馈里有主题相关内容，并且信息通过观众操作被获得或改变，不是空效果或静态说明 |
| 资产使用克制 | 5 | 只使用相关 atomic/example，不变成组件组图或设备清单 |
| 视觉 DNA | 5 | 专业展项方案草图感，少量标注，非海报/舞美/3D 渲染/UI 图，蓝色/青色不过量 |

## 一票否决

出现以下任意情况，即使画面好看也不得超过 69 分：

- 没有人物，或人物只是装饰。
- 人物风格明显漂移：有发型、五官、复杂服装、写实游客或不同人物系统。
- 看不出怎么玩，只能靠文字解释。
- 用户指定的小屏幕反馈被换成独立大屏、悬浮卡或说明条。
- 投影、探照灯、扫描灯、定向声音或感应设备没有合理工作距离、路径或范围。
- 投影源、光路方向和投影落点不一致，或把交互箭头误当投影光路。
- 同一投影源出现多个冲突出光口，例如前镜头发光但光束从顶部孔、机身表面或另一个槽口发出。
- 投影目标面不在镜头光学轴线前方，例如为了补充内容把投影画在右侧墙面，但投影机并没有朝向那里。
- 投影画面没有合理介质，像空中悬浮 UI，或落在不面向光束的墙/屏幕上。
- 投影输出虽然光轴成立，但无论题材都默认变成外接屏幕/右侧幕布，丢失了投影映射到物体、模型、桌面、地面或半透明介质的可能性。
- 人物够不到操作点，儿童操作成人高度控件。
- 固定展项的把手、手轮、拉杆、滑块、按钮、投影源、屏幕、传感器、支架或设备外壳看起来从人物手、手臂、身体、衣服或儿童身体上长出来，或人物身体被当成固定硬件安装面。明确的 VR、手持控制器、可穿戴传感器或互动服不适用此条，但必须有可读穿戴/握持结构。
- 普通互动被无故画成顶天立地轨道、全高立柱、龙门架、吊架或大型桁架。
- 屏幕、投影或灯光只有通用蓝色 UI / 空光效，没有主题信息。
- 主题信息、参数、流程或原理说明被完整贴在装置上；去掉人物动作后信息仍然完整成立，像静态展板或策划说明图。
- 环境光、信号强度、系统状态等被画成设计者视角的参数牌、诊断仪表或说明模块，而不是观众可操作的任务和即时反馈。
- 选择、识别、复原、比较、拆解、分层或时间推进只靠一个静态对象表达，3 秒内看不懂玩法。
- 整张图被同一种蓝色/青色支配，主题内容、反馈层级和标注无法区分。
- 画面像组件库、设备清单、分镜图、海报、PPT 信息图、技术布线图或游戏插画。

## 回归记录格式

建议把每次测试结果记录在 `outputs/validation-runs/` 中：

```text
date: YYYY-MM-DD
skill_version_or_change: {本轮改了什么}
case_id: {T01...}
result_image: {路径；dry-run 时写 none}
score: {0-100}
pass_status: pass / partial / fail
blocking_issues:
- ...
next_fix:
- ...
```
