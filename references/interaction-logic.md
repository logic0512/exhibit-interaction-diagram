# 交互逻辑

## 一句话玩法

生成前先把创意转成一句：

某个参与者在某个位置做某个动作，触发某个装置或系统，在某个区域产生某种反馈，形成某种体验结果。

## 必须画清的四件事

### 人

谁在参与，站在哪里，朝向哪里。

### 触发

人做了什么动作：摸、挥、踩、靠近、转动、拿取、放置、扫描、协作、走入。

### 反馈

变化在哪里发生：屏幕、投影面、灯光、声音、机械件、地面、空间、道具、墙面。

### 结果

观众看到或感受到什么：影像生成、内容解锁、灯光变化、声音出现、路径点亮、物体运动、环境沉浸。

## 玩法证据

玩法不是一句文字说明，必须在画面里有足够的“证据”让人看懂。

- 选择/比较：至少出现 2-4 个可选对象、选项位置或对比状态。
- 扫描/识别：要有被识别对象、识别区域和识别后的反馈；如果只有一个对象，必须画出识别前后差异。
- 复原/拆解/分层：要有多个层级、阶段或前后状态；不要只放一件静态展品。
- 时间推进：要有时间节点、连续位置、胶片/轨道/刻度/路径或前后变化。
- 协作：要有多个参与者各自可执行的动作或输入点，而不是多人围观同一个按钮。

如果一个对象无法单独说明“为什么要互动”，就增加对象数量、状态层级或动作阶段；不要用长文字解释弥补。

## 画面连接方法

- 用人物动作连接触发点。
- 用箭头、虚线、光束、波纹或视线连接触发和反馈。
- 用反馈区域的发光、变化图形或小标注说明结果。
- 空间关系优先于技术原理，少画后台系统。

## 判断顺序

1. 这是不是单人、双人、多人或围观型互动。
2. 主要动作发生在手、脚、身体靠近、视线、道具还是空间路径上。
3. 反馈是屏幕型、投影型、灯光型、声音型、机械型、环境型还是混合型。
4. 需要哪几个载体组件才能让玩法清楚。
5. 哪些组件会造成干扰，应该删掉。
6. 是否需要多个对象、多个状态或连续位置才能支撑玩法。

## 禁止

- 不要画成设备布线图。
- 不要画成 UI 流程图。
- 不要只表现氛围，不表现玩法。
- 不要只画装置造型，不画人怎么用。
- 不要为了完整而塞入所有载体。
- 不要把选择、识别、复原、比较或时间推进画成单一静态物件。

## 提示词短句

The image must clearly show the interaction chain: participant action -> trigger point or sensing area -> visible feedback area -> experiential result. Make this chain legible through pose, arrows, dashed zones, beams, waves, gaze direction, and a few short handwritten labels.

If the gameplay depends on selection, comparison, recognition, reconstruction, layered reveal, disassembly, or timeline progress, show enough visual evidence: multiple selectable objects, multiple states, layered stages, or continuous positions. Do not rely on a single static object unless it visibly changes.
